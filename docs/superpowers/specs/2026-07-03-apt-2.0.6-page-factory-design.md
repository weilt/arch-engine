# APT 2.0.6：页面工厂（开发冻结 → 批量实现）

**日期:** 2026-07-03  
**状态:** 已批准（brainstorming 2026-07-03）  
**关系:** 扩展 `2026-06-25-apt-2.0-autonomous-loop-design.md`（2.0.5）、`2026-07-02-v0-page-handoff-design.md`（v0 适配器与 design-sync）  
**方案:** 两阶段工厂 — Phase A 开发逐页冻结 logic；Phase B `plan-from-spec` → `implement-plan` 串行实现  

---

## 1. 背景与目标

### 1.1 动机

APT 2.0.5 + v0 handoff（`design-sync --adapter v0`）已具备：设计入库、`query_design(page:)`、契约/arch 寻址、`/feature` 闭环、`/apt-goal` 自主编排。

现实缺口：

| 缺口 | 说明 |
|------|------|
| **PM 不写双文件** | `page.manifest.json` / `page.logic.md` 不应要求 PM 产出或审到正确 |
| **逻辑未冻结就开发** | v0 视觉与真实业务逻辑易漂移；需 **全页逻辑人工说清楚** 后才能开工 |
| **多页编排散** | 单页 `/feature` 可跑，但 10+ 页缺标准 rollup plan 与 Task 模板 |
| **缺 API 时流程不显** | `report_missing` 会停，但 plan 未固化「先后端再前端」三段式 |

### 1.2 2.0.6 定位

**工作名：** 页面工厂 — 在**单语言栈**（现有 Java 后端 + 单前端）下，把「v0 成品 + PM 设计文档」变成可批量开发的冻结页面集，再经 plan 串行实现并入库。

| 纳入 2.0.6 | 不纳入（后续版本） |
|------------|-------------------|
| 开发主导 Phase A handoff Skill | **3.0 多语言 / 多栈** |
| `_pages.md` 进度与冻结门禁 | v0.dev API 自动拉取 |
| rollup spec + 单页 Task 模板 | `apt-goal` 自动读页队列（可 2.1） |
| Phase A 门禁检查（Skill/脚本） | 合并 `.ai/design` 与 `.ai/arch` |
| 修订 v0 叙事（开发写 logic） | verify 自动 diff logic 与实现 |

### 1.3 成功标准

1. PM 仅交 **v0 成品 + 设计文档**；开发用 **`apt-v0-handoff`** 逐页建目录、写双文件、`approved`。
2. **`designs/v0/_pages.md` 全部 `approved`** 前，**禁止** `/plan-from-spec` 进入全页实现（硬门禁）。
3. Phase B：`plan-from-spec`（rollup spec）→ 用户确认 → `implement-plan` 每页串行；接口 **仅** 经 `query_contract` / `search_arch` / `query_arch` 选定；无则新建并 `register_contract` / `refresh_asset`。
4. 全链路以 **`/verify` PASS** 为交付门禁（与 2.0.5 `loopDone` 一致）。
5. 2.0.6 完成后，单人 + `/apt-goal` 可驱动「页面工厂」完整路径（Phase A 可多会话，Phase B 一次 plan）。

## 2. 已确认决策（brainstorming）

| 项 | 选择 |
|----|------|
| 双文件作者 | **开发**对照 v0 + PM 文档核对后写入 |

| 新斜杠命令 | **不新增**；`apt-v0-handoff` 为纯 Skill |
| 与 v0 spec §4 PM 伴侣 | **降级为可选草稿**；主路径为开发 handoff |

## 3. 两阶段模型

```text
Phase A — 页面逻辑冻结（开发，最难，逐页人工）
  输入：v0（tsx/链接/截图）+ PM 设计文档
  输出：designs/v0/<page-id>/{manifest,logic,page.tsx?}
  进度：designs/v0/_pages.md
  结束：全部 approved + 批量 design-sync

Phase B — 批量开发
  输入：rollup spec（基于 _pages.md + 各页 logic）
  编排：plan-from-spec → 确认 → implement-plan（串行）→ verify
```

---

## 4. Phase A：开发逐页 Handoff

### 4.1 PM 交付（仅素材）

| 交付物 | 说明 |
|--------|------|
| v0 成品 | 链接、导出 `page.tsx`、截图；可平铺 `designs/v0-inbox/` |
| 设计文档 | PRD、流程、权限、业务规则（仓库内 `docs/` 等） |

PM **不**负责：`page.manifest.json`、`page.logic.md`、`design-sync`。

### 4.2 目录约定

```text
designs/v0-inbox/          # 可选：PM 平铺素材
docs/...                    # PM 设计文档
designs/v0/_pages.md        # 开发维护：页面清单与冻结进度

designs/v0/<page-id>/
  page.manifest.json
  page.logic.md
  page.tsx                  # 推荐：v0 导出
  preview.html              # 可选
```

### 4.3 `designs/v0/_pages.md`

开发维护的 SSOT 进度表（Markdown 表格）：

| 列 | 含义 |
|----|------|
| `page-id` | kebab-case，与 manifest.id 一致 |
| `title` | 中文标题 |
| `handoff` | `pending` \| `wip` \| `done` — 双文件是否写完 |
| `approved` | `no` \| `yes` — 开发认定已按 PM 设计说清楚 |
| `synced` | `no` \| `yes` — 是否已 design-sync |
| `notes` | 待对文档、权限等待办 |

### 4.4 Skill：`apt-v0-handoff`（无新 `/` 命令）

**唤起：** Codex `apt-v0-handoff`；Cursor 等通过 `.agents/skills/apt-v0-handoff/`。

**每页一轮（禁止批量糊 logic）：**

1. 确认本页 `page-id`（从 v0 标题/路由 + 文档推断，与用户确认）。
2. `query_design(scope: global)` — 记录语义组件 id，禁止臆造。
3. 对照 v0 控件 + PM 文档，创建/更新 `designs/v0/<page-id>/`。
4. 写入 `page.manifest.json`、`page.logic.md`（`status: draft` → 核对后 `approved`）。
5. logic **必须**含：操作明细表、主流程、状态、§依赖（API **意向名** + 语义组件 id）。
6. 推荐拷入 `page.tsx`；单页 `design-sync --adapter v0 --source designs/v0/<page-id>`。
7. `query_design(page:)` 自检；更新 `_pages.md` 对应行。
8. **禁止**写 `src/` 业务代码。

内联规则来自 `templates/v0-visual-handoff-prompt.md`（字段、pageType 枚举、禁止项）。

### 4.5 Phase A 结束门禁（硬条件）

以下 **全部** 满足才允许 Phase B：

1. `_pages.md` 中 **每一行** `approved = yes`。
2. 每页存在 `page.manifest.json`、`page.logic.md`。
3. 执行 `design-sync --adapter v0 --source designs/v0` 成功。
4. 抽检每页 `query_design(page:)`：`gaps` 不含 `manifest-not-approved`、`missing-logic`。

**门禁实现（2.0.6）：** Skill 步骤 + 可选 CLI/脚本 `check-v0-freeze`（读 `_pages.md` 与目录，exit 1 若未满足）。`/plan-from-spec` 模板首段要求先通过门禁。

## 5. Phase B：rollup spec 与批量实现

### 5.1 Rollup spec

路径：`docs/superpowers/specs/YYYY-MM-DD-<product>-pages-rollout-spec.md`

由开发/TL 在 Phase A 完成后撰写（非 PM）。骨架见 `templates/page-rollout-spec.md`（2.0.6 新增）。

必含：

- 声明 **Phase A 已全部 approved**（附 `_pages.md` 快照或 commit）。
- 页面清单与 `feature` 域划分。
- 跨页约束（共用 layout、权限、路由前缀）。
- 每页引用 `designs/v0/<id>/page.logic.md` 路径（实现时经 `query_design` 读，禁止直读 `.ai/design/` 猜）。

### 5.2 编排

```text
check-v0-freeze（或 apt-v0-handoff 门禁步骤）PASS
  → /plan-from-spec <rollup-spec>
  → 用户确认 plan
  → /implement-plan（每页 Task 串行子 Agent）
  → /verify
  → /finish-feature（若 FAIL）
```

与 `/apt-goal`：`phase` 在 Phase A 期间可为 `page_handoff`（文档约定）；Phase B 走现有 `plan_from_spec` → `implement_plan` → `verify` `nextAction` 链。

### 5.3 每页 Task 标准拆分（plan Part 2 模板）

每个 `page-id` 至少下列结构（小页可合并 Task，但须覆盖能力）：

| Task | 名称 | 内容 |
|------|------|------|
| **B1** | 依赖与接口 | `query_design(page:)` 读 logic；对每个 API 意向名：`query_contract` → `search_arch` → `query_arch` |
| | | 无命中：`query_impact` / `query_ontology` 定落点 → 在合适模块 **新建** API/client |
| | | 新建后：`register_contract`（TS）/ `refresh_asset`（arch） |
| **B2** | 前端页面 | `query_design` global + 语义组件；按 `refs/<id>.tsx` + bindings 实现 |
| | | **以冻结 logic 为 SSOT**；偏离 → 先改 logic 并重新 sync，不得静默漂移 |
| | | `gaps` 含 blocking 项 → `report_design_gap`，停 UI |
| **B3** | 页级闭环 | `register_ui_pattern`；本 Task 触及的 arch `refresh_asset`；子 Agent 微闭环 |

**接口规则：** 禁止凭 logic 或 v0 mock 臆造签名；以 MCP 寻址结果为准。

### 5.4 与 `/feature` 的关系

- 单页、无 rollup 时仍可用 `/feature`（Phase A 未完成时仅允许非 UI 或单页 handoff）。
- **全页批量**必须用 Phase B（plan-from-spec），避免主 Agent 无 plan 漂移。
