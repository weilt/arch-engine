# APT 2.0：自主闭环（Solo Autonomous Loop）

**日期:** 2026-06-25  
**状态:** 待用户审阅  
**关系:** 扩展 APT 1.x 命令与 MCP；衔接 `/feature`、`/implement-plan`、`/verify`、子 Agent 编排 spec  
**方案:** MCP 状态中心 + `/apt-goal` 外层编排 + AI brainstorming 风险分级  

---

## 1. 背景与目标

### 1.1 动机

APT 1.x 已具备：契约/架构/设计 MCP、7 斜杠命令、子 Agent 串行实现、`/verify` 验收门禁。缺口在于：

- **跨会话持续推进**：平台 `goal` / `loop` 不会自动走 APT 流程；裸写 goal 易跳过 `query_contract`、`/verify`、spec 审批。
- **进度不可机器读**：`.apt/orchestration/` 为 gitignore 本地账本，无统一 `loopDone` 判定。
- **Brainstorming 依赖人**：superpowers brainstorming 需人逐条回答；自主交付需 AI 代劳 + 高风险人批。

### 1.2 2.0 定位（非团队版）

**工作名：** APT 自主闭环 — 单人 + 多子 Agent 交付**完整产品**，不实现团队 Hub、看板、owner 分工、共享向量库。

| 保留 | 不做（YAGNI / 后续） |
|------|----------------------|
| `/apt-goal` 强制外层编排 | APT Hub、实时看板 |
| `query_project_status` + `/current-status` | 团队进度同步 |
| `/auto-brainstorm` + 风险分级人批 | 自建平台 `/goal` |
| `status.json` / `approvals.json` 提交 Git | 2.0 远程向量库（可留配置占位） |
| 可选平台 `loop` + hook 配方 | 无人值守跳过 verify |

### 1.3 成功标准

1. 用户执行 **`/apt-goal <产品目标>`**，Agent **仅编排**，按 `query_project_status.nextAction` 依次驱动子流程，**禁止**跳步与 inline 实现。
2. **`loopDone === true`** 当且仅当：plan 全 Task DONE + **最近 verify PASS** + arch audit 空 + 无审批阻塞 + 无 blockers。
3. **`/current-status`** 与 **`query_project_status`** 输出一致、可机器读。
4. **`/auto-brainstorm`**：低风险 spec 自动批准；高风险暂停等人批（决策 D）。
5. 文档说明：优先用 **`/apt-goal`** 替代裸平台 goal；可选 `loop` + `apt-status` 做多会话续跑。

## 3. `/apt-goal` 与平台 goal 的关系

### 3.1 能否让 IDE goal 自动走 APT？

**不能（默认）。** 平台 `goal` / `loop` 只负责「再跑一轮 Agent」，**不知道** APT 阶段与 `nextAction`。裸 goal 易直接写代码。

### 3.2 `/apt-goal` 是什么？

**APT 自有外层编排命令**（第 10 个斜杠命令），**取代**在业务项目上使用裸平台 goal 作为主入口。

```text
用户: /apt-goal 做出 XXX 完整产品

/apt-goal 主 Agent（只编排，禁止实现代码）
  │
  ├─ 写入 .apt/goal.md（产品目标 SSOT）
  ├─ 刷新 .apt/status.json
  │
  └─ 循环（单会话内 prompt 强制，或多会话靠平台 loop 续跑）:
        1. query_project_status（必须）
        2. 若 loopDone → 输出交付摘要，结束
        3. 若 blockers → 停住报告（高风险未批等）
        4. 仅按 nextAction 执行**唯一**对应子流程（见 §3.3）
        5. 子流程结束 → 刷新 status.json → 回到 1
```

### 3.3 `nextAction` → 子流程映射（硬规则）

| nextAction | 执行 |
|------------|------|
| `auto_brainstorm` | 等同 `/auto-brainstorm`（附 goal.md 上下文） |
| `await_spec_approval` | **停止**，提示人审 spec |
| `plan_from_spec` | 等同 `/plan-from-spec` |
| `implement_plan` | 等同 `/implement-plan`（子 Agent 串行） |
| `feature` | 等同 `/feature`（仅无 spec 的退路） |
| `verify` | 等同 `/verify`，写入 `.apt/verify/latest.md` |
| `finish_feature` | 等同 `/finish-feature` |
| `start_init` | 提示 `start-init`，标记 blocked |
| `none` | loopDone，结束 |

**禁止：** 在 `implement_plan` / `feature` 阶段主 Agent inline 编码；跳过 `verify` 宣称完成。

### 3.4 与平台 loop 的配合（可选加速器）

`/apt-goal` **已含**单会话内循环逻辑。跨会话 / 关标签续跑时，推荐：

```text
/loop until apt-status reports loopDone: /apt-goal --continue
```

或 Cursor `stop` hook 读 `apt-status --json`，`followup_message` 注入「继续 /apt-goal --continue」。

- **`/apt-goal`**：流程 SSOT + 每轮该跑什么  
- **平台 loop**：进程/scheduling 层，**不**定义业务步骤  

Codex Skill 名：`apt-goal`（`$apt-goal`）。

## 4. 架构

### 4.1 数据流

```text
.apt/goal.md          ← /apt-goal 写入产品目标
.apt/status.json      ← 各阶段刷新（提交 Git）
.apt/approvals.json   ← spec 审批（提交 Git）
.apt/verify/latest.md ← /verify 报告（提交 Git）
.apt/orchestration/   ← Task 账本（仍 gitignore）

query_project_status ← aggregate 读上述 + plans/specs/progress
        │
        ├── /current-status（人读）
        ├── apt-status --json（hook/loop）
        └── /apt-goal 每轮决策
```

### 4.2 阶段状态机

| phase | 判定 |
|-------|------|
| `idle` | 无活跃 spec/plan/orchestration |
| `brainstorming` | draft spec 或 auto-brainstorm 进行中 |
| `spec_pending_approval` | 高风险 spec，未 approved |
| `planning` | spec 已批，无 approved plan |
| `implementing` | progress.md 有未 DONE Task |
| `verifying` | 全 Task DONE，verify 未 PASS |
| `done` | 满足 §5 loopDone 全条件 |
| `blocked` | 缺 agent-init/start-init、verify BLOCKED 等 |

### 4.3 新增 MCP：`query_project_status`（第 16 工具，只读）

```typescript
interface ProjectStatus {
  phase: string;
  loopDone: boolean;
  nextAction: string;
  goal?: string;              // .apt/goal.md 摘要
  activeSpec?: string;
  activePlan?: string;
  specRisk?: "low" | "high";
  specApproval?: "pending" | "approved" | "auto_approved";
  tasks?: { total: number; done: number; blocked: number };
  lastVerify?: { path: string; result: "PASS" | "FAIL" | "BLOCKED" | "none" };
  blockers: string[];
  summary: string;
}
```

`query_project_status` 读 `status.json` 快照并**重算校验**；与持久化不一致时以重算为准并可选回写。

### 4.4 模块（mcp-server）

| 文件 | 职责 |
|------|------|
| `src/status/types.ts` | 类型 |
| `src/status/aggregate.ts` | phase / loopDone / nextAction |
| `src/status/risk.ts` | spec 风险规则 |
| `src/status/verify-parse.ts` | 解析 verify 报告 |
| `src/cli-status.ts` | `apt-status --json` |
| `tests/status/*.test.ts` | 单测 |

## 5. `loopDone` 硬条件（全部满足）

1. `.apt/goal.md` 存在且 activePlan 关联该交付。  
2. plan Part 2 **全部 Task = DONE**（`progress.md`）。  
3. **最近 verify**（`.apt/verify/latest.md`）**result = PASS**。  
4. `audit_arch_changes` 四类皆空（verify PASS 已含此检查；status 重算时 double-check）。  
5. 无 `spec_pending_approval`、无 `blockers`。  
6. 含 UI 时 verify 已覆盖 design audit 或 status 无 design blocker。

**哲学：** 自主项目的兜底是 **verify PASS**；没有 verify 不能声称「按预想执行」。

## 6. AI Brainstorming（`/auto-brainstorm`）

### 6.1 流程

1. 读 `.apt/goal.md`（无则读用户参数）。  
2. 执行 superpowers brainstorming 逻辑，**提问与方案选择由 AI 完成**（非人）。  
3. 写 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md`。  
4. 运行 `risk.ts` 分级。  
5. **low** → `approvals.json` 记 `auto_approved`，可进 `plan-from-spec`。  
6. **high** → spec `status: draft`，`phase=spec_pending_approval`，**停**等人说「批准 spec」。  
7. 刷新 `status.json`。

### 6.2 高风险触发（规则引擎，v1 无 ML）

- 变更 `mcp-server` / 新 MCP 工具  
- 新对外契约 / 破坏性 API  
- `arch-engine` 或 arch 管线变更  
- 预估 >8 文件或 >2 新服务（启发式）  
- 用户可在 spec frontmatter 手动标 `risk: high`

## 9. 错误处理

| 场景 | 行为 |
|------|------|
| 无 `.ai/db.json` | `blocked`，提示 `agent-init` |
| 无 `last-scan.json` | `blocked`，提示 `start-init` |
| high risk spec 未批 | `await_spec_approval`，`/apt-goal` 停住 |
| verify FAIL | `nextAction=finish_feature`，`loopDone=false` |
| progress 与 plan 不一致 | `blockers` 列出，需人修 |
| 单会话 context 过长 | 建议平台 `loop` + `/apt-goal --continue` |
